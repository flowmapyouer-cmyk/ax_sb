# Challenge — 스펙 리스크 의사결정 요청서 작성

## Role

- **목적:** 의사결정권자(C-Level/대표)가 문서를 처음부터 끝까지 읽지 않아도, 상단만 보고
  "지금 뭘 결정해야 하는지"를 30초 안에 파악하게 한다. 나머지는 필요할 때만 펼쳐 읽는
  참고 자료로 둔다.
- **이 스킬이 하는 일:** 10주 MVP 스프린트의 성공적인 출시를 방해하는 [TODO]와 모순(Conflict)을
  감지하고, Blocker(지금 결정)/Warning(이번 스프린트 중 결정)/참조(결정 불필요, 참고만)
  3단계로 분류해 '스펙 리스크 의사결정 요청서'를 작성한다.
- **다른 스킬과의 구분:** PRD 생성 스킬에 쓰이는 우선순위 매트릭스(P1/P2/P3, 대상: PM/개발자,
  "언제 개발할지")와는 목적과 대상이 다르다. 이 스킬은 "지금 당장 대표가 결단 내려야 할 것이
  뭔지"에 집중한다. 두 스킬을 혼용하지 말 것.

## Trigger Context

- **슬래시 커맨드:** `/의사결정`
- **키워드:** 비정형 텍스트 또는 파일 안에 슬랙, CS, 이메일, 회의, 메모가 포함되었을 때
- **공통 트리거:** 비정형 텍스트(슬랙/CS/이메일/회의/메모) 또는 파일 업로드 후 "보고 하고
  싶어", "보고서 작성하고 싶어", "내가 팀장님이라면 무엇부터 결정해야 하는지 알고 싶어",
  "리스크 요약해줘" 등의 간편 프롬프트 입력 시
- **입력 처리 원칙:** 이 스킬은 독립형(standalone)으로 설계되어, 매번 입력된 원문을 처음부터
  재분석한다. 기존에 만들어진 다른 산출물(PRD 문서, 결정 로그 등)을 참조하지 않으며, 어떤
  원문·더미 데이터가 들어와도 동일한 판정 로직으로 동작해야 한다. 따라서 출력물의 `[케이스 N]`
  번호는 **이번 입력 원문 안에서 새로 매기는 번호**이며, 다른 문서의 케이스 번호 체계와는 무관하다.

## Operational Rules

0. **[필수 선행] 판정 기준 로드:** 이 스킬을 실행하기 전에 반드시
   [`references/challenge-matrix.md`](references/challenge-matrix.md) 전체를 읽고, 그 문서의
   A~F축(우선순위 시그널)·이진화 규칙(P1→이번 스프린트급, P2/P3→다음 스프린트급, P3는 계산
   대상 완전 제외)·G축(구현 리스크/재작업 비용, 키워드 사전+문맥 결합 판단)·Blocker/Warning
   연산식을 그대로 따를 것. 아래 1번 항목은 그 매트릭스의 결과를 요약한 것일 뿐이며, 세부 판단
   기준은 `challenge-matrix.md`가 항상 우선한다.

1. **리스크 등급 연산 Matrix 적용 (challenge-matrix.md 요약):**

   - **🚨 Blocker — 지금 결정 필요:** 이번 스프린트급(P1) 스펙인데 정책/수치가 미정이거나
     상반된 요구(Conflict)가 충돌하는 경우 (출시 지연 직결 요인)
   - **⚠️ Warning — 이번 스프린트 중 결정 필요:** 다음 스프린트급(P2/P3 통합) 스펙 중 미결
     사항이 있거나 예외 처리 정책이 비어있는 경우, **또는 G축 기준(신규 기술/전문 영역 요구·
     외부 솔루션 의존·구현 방식 상반)에 해당해 재작업 비용이 클 것으로 판단되는 경우** (단,
     G축은 키워드 매칭만으로 단독 확정하지 않는다 — 반드시 F축 미결 또는 Conflict 동반 여부를
     함께 판단하며, 방향이 이미 확정되고 일정만 급한 경우는 리스크가 아니라 일정 압박으로
     취급한다)
   - **📌 참조 — 결정 불필요, 참고만:** Blocker/Warning 대상이 아닌, 원문에서 명시적으로
     후순위·제외로 결정된 항목 (challenge-matrix.md의 P3 = 계산 대상 제외 규칙 적용). 이미
     결정된 사항이라 압박하는 톤이 아니라 가볍게 남겨둔다. C-Level은 이 섹션을 읽지 않고
     넘어가도 무방하다는 점을 문서 안에 명시한다.
     - 참조 예시:
       - 색약 모드: 이번 릴리스 범위에서 제외로 결정됨 (원문 근거: "색약 모드는 이번 릴리스
         범위에서 제외")
       - 매출 그래프 주별 단위 / 기간 필터: 추후 적용으로 결정됨

2. **원문 그대로 병기, 결론 임의 도출 금지:** 대립하는 문구를 날것 그대로 병기하여 의사결정에
   참고할 수 있도록 원본을 유지하라. (단, 📌 참조 섹션은 원문을 그대로 적용하지 않는다.) **특히
   "이 기능은 재정적으로 문제가 될 것이다", "이 방식이 더 낫다" 같은 AI 자체 추정 근거로 결론이나
   권고를 만들어내지 말 것** — 원문에 없는 비즈니스 맥락(실제 예산, 고객사 사정 등)은 AI가 알 수
   없으므로, 그 판단은 전적으로 의사결정권자의 몫으로 남겨둔다.

3. **Google Docs 붙여넣기 최적화 포맷 — 문자 그대로 출력:** 이 스킬의 결과물은 사용자가
   그대로 복사해서 Google Docs에 **Ctrl+V로 붙여넣는 것**을 전제로 한다. Google Docs의
   마크다운 붙여넣기는 제목(`#`~`######`)·굵게/기울임·리스트·체크박스(`- [ ]`)는 실제
   스타일로 변환하지만, **마크다운 표(`| --- |`)는 표로 변환되지 않고 `|` 기호가 그대로
   텍스트로 남는다** — 그래서 이 스킬은 표를 쓰지 않고 전부 굵은 소제목 + 불릿 리스트로
   대체한다. 콜아웃·토글도 Google Docs 붙여넣기로 만들 수 없으므로 아예 쓰지 않는다.
   - **Executive Summary 필수:** 문서 맨 위에 Blocker 전체를 체크리스트로 요약한
     "⚡ 지금 결정해야 할 것" 섹션을 반드시 넣는다. C-Level이 이 섹션만 읽고 넘어가도
     무엇을 결정해야 하는지 알 수 있어야 한다.
   - **제목 레벨로 긴급도 표현:** Blocker는 `#`(그룹)/`##`(케이스), Warning은 `#`(그룹)/
     `###`(케이스), 참조는 `####`만 사용해 위에서 아래로 갈수록 시각적 비중이 작아지게 한다.
   - **섹션 라벨에 "결정 필요/참고만" 명시:** 각 그룹 제목에 "지금 결정 필요", "이번 스프린트
     중 결정 필요", "참고만 하면 됨(결정 불필요)"을 반드시 병기해 제목만 보고도 액션 여부를
     알 수 있게 한다.
   - `[TODO] ...` 항목: `- [ ]` 체크박스 문법 그대로
   - 원문 내 상반된 요구사항(Conflict) 병기: `- {출처}: "{원문 인용}"` 불릿 리스트로 (표 금지)

## Output Format (Google Docs Copy-Paste Optimized)

**중요:** 아래 마크다운을 문자 그대로 출력한다. 설명하듯 서술하는 문장형 bullet이 아니라
`- [ ]` 체크박스, `#`~`####` 제목, 굵은 소제목(`**...**`) 문법 그대로 써야 한다. 마크다운
표(`|---|`)나 `<aside>`/`<details>` 같은 HTML 태그는 절대 쓰지 않는다 (Google Docs
붙여넣기 시 표는 깨지고 HTML 태그는 글자 그대로 노출된다). 코드 블록으로 감싸지 말고 실제
마크다운으로 바로 출력할 것.

```markdown
# 🔴 스펙 리스크 의사결정 요청서

소속: {팀/조직명 — 원문에서 알 수 없으면 사용자에게 확인하거나 플레이스홀더 유지}
목적: 10주 MVP 출시 지연 방지를 위한 최우선 의사결정 과제 리스트업

## ⚡ 지금 결정해야 할 것 (Executive Summary)
- [ ] 🚨 Blocker {N} — {케이스 타이틀}: {한 줄 요약}
- [ ] 🚨 Blocker {N} — {케이스 타이틀}: {한 줄 요약}

**읽는 법:** 🚨 Blocker만 지금 결정하면 됩니다. ⚠️ Warning은 이번 스프린트 중에,
📌 참조는 이미 결정된 사항이라 읽지 않고 넘어가셔도 됩니다.

---

# 🚨 지금 결정 필요 — Blocker

## Blocker {N}번 - {케이스 타이틀}

**핵심 의사결정 요청 사항**
- [ ]  [TODO] {누구}에게 {무엇} 확인

**원문 내 상반된 요구사항 (Conflict)**
- {출처1, 예: 대표 슬랙}: "{원문 인용}"
- {출처2, 예: 실무자 이메일}: "{원문 인용}"

#### 판단 근거 및 일정 임팩트
{판단 근거 — 원문 근거 + challenge-matrix.md 판정식에 기반한 서술만. 일반 문단으로 서술.}

---

(위 "## Blocker {N}번" 블록 전체를 Blocker 케이스 수만큼 반복하고 `---`로 구분한다.
번호는 1번부터 순서대로 매긴다. Conflict가 한 곳뿐이면 불릿 1개만 채운다.)

# ⚠️ 이번 스프린트 중 결정 필요 — Warning

### Warning {N}번 - {케이스 타이틀}

**핵심 의사결정 요청 사항**
- [ ]  [TODO] {누구}에게 {무엇} 확인

**원문 내 상반된 요구사항 (Conflict)**
- {출처1}: "{원문 인용}"
- {출처2}: "{원문 인용}"

---

(Blocker와 동일한 구조를 Warning 케이스 수만큼 반복. 번호는 Warning 안에서 1번부터
새로 매긴다.)

# 📌 참고만 하면 됨 — 결정 불필요 (참조)

1. {항목}: {결정된 내용} (원문 근거: "{인용}")
2. {항목}: {결정된 내용} (원문 근거: "{인용}")
```

## Working procedure

1. `references/challenge-matrix.md`를 전체 읽는다 (0번 규칙, 매번 새로 읽을 것).
2. 사용자가 제공한 비정형 원문(슬랙/CS/이메일/회의록/메모)을 처음부터 끝까지 읽으며, 기능/이슈
   단위로 후보 항목을 추출한다.
3. 각 후보 항목에 대해 A~F축 신호를 식별하고, 매트릭스의 "복합 판단" 원칙에 따라 P1/P2/P3를
   종합 판정한다. P3(명시적 후순위/제외)는 계산 대상에서 제외하고 📌 참조로 분류한다.
4. G축(구현 리스크) 여부를 사전 등록된 키워드(다국어, AI/추천, 음성인식, OCR, 생체인증, 색약 등)
   기준으로만 판단하고, 반드시 F축 미결 또는 Conflict 동반 여부와 결합해서 최종 확정한다.
   키워드만 있고 방향이 확정+일정만 급한 경우는 리스크로 잡지 않는다.
5. Blocker/Warning 연산식을 적용해 최종 등급을 확정한다.
6. 먼저 확정된 Blocker 전체를 한 줄씩 요약해 `## ⚡ 지금 결정해야 할 것` 체크리스트를
   작성한다 (문서 맨 위, "읽는 법" 안내 문구 포함).
7. Output Format에 맞춰 본문을 출력한다: `# 🚨 지금 결정 필요 — Blocker` 그룹 아래 각
   케이스를 `##`로, `# ⚠️ 이번 스프린트 중 결정 필요 — Warning` 그룹 아래 각 케이스를
   `###`로, 마지막에 `# 📌 참고만 하면 됨 — 결정 불필요`를 배치한다. "판단 근거 및 일정
   임팩트"는 모든 그룹에서 `####` 제목 + 일반 문단으로 쓴다. 표(`|---|`)와 콜아웃/토글
   문법(`<aside>`, `<details>` 등)은 절대 사용하지 않고, Conflict 비교는 불릿 리스트로
   대체한다. 그룹 사이·케이스 사이는 `---`로 구분한다.
8. 결론이나 권고(예: "이 기능은 재정적으로 문제가 될 것이다")를 AI가 자체적으로 만들어내지
   않았는지 다시 한번 검토한다. 판단 근거는 항상 원문 인용 + 매트릭스 판정식에만 근거해야 한다.
9. 출력 결과는 그대로 복사해서 Google Docs에 Ctrl+V로 붙여넣으면 제목·체크박스·굵은 텍스트가
   즉시 정상 렌더된다. 별도의 파일 저장이나 Import 절차가 필요 없다.
