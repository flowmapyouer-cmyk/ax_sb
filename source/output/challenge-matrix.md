# Challenge 스킬 전용 리스크 매트릭스 (Blocker / Warning) — 독립형(Standalone)

> **범위 주의**: 이 문서는 [priority-matrix.md](priority-matrix.md)(PRD 생성 스킬용, P1/P2/P3
> 3단계)와 **절대 혼용하지 않는다.** 3번째 스킬 — "스펙 리스크 의사결정 요청서(Challenge)" —
> 전용이며, 목적도 다르다.
>
> - priority-matrix.md → PRD 문서용, "이 기능을 언제 개발할지" (P1/P2/P3, 대상: PM/개발자)
> - challenge-matrix.md(본 문서) → 의사결정 요청서용, "지금 당장 대표가 결단 내려야 할 것이 뭔지"
>   (Blocker/Warning 2단계, 대상: C-Level/대표)
>
> **독립형으로 설계된 이유**: 시연회에서 지금 데이터(raw_requirements.v2.txt)가 아니라 더미 데이터를
> 쓸 수 있으므로, priority-matrix.md 파일을 별도로 참조하지 않고 이 문서 하나만으로 우선순위
> 판정부터 리스크 등급 산출까지 끝까지 동작해야 한다. 그래서 A~F축을 요약이 아니라 **전문**으로
> 인라인했다.
>
> 이 문서는 판정 로직(매트릭스)만 담는다. 스킬에 넣을 프롬프트 문구·산출물 출력 포맷(참조 섹션
> 표기 방식, Notion 템플릿 등)은 별도 파일에서 다룬다.

## 1. 상위 축 A~F (전문, 인라인)

### 우선순위 시그널 6대 축

| 축 | 정의 | 신호(키워드) |
|---|---|---|
| **A. 명시적 긴급 키워드** | 화자가 직접 급함을 표현 | 꼭, 반드시, 필수, 즉시 |
| **B. 외부 일정/데드라인** | 서비스 외부 일정과 결부 | 이번 스프린트, 데모 다음주, 이번 분기, 긴급 미팅 |
| **C. 이해관계자 권위/출처** | 발화자의 직책·전문성 | 사장님/경영진 지시, 대표 직접 요청, 보안 컨설턴트 권고 |
| **D. 수요/불만 볼륨** | 요청·불만의 양적 신호 | VoC 많음, 불만 다수, 요청 증가 |
| **E. 명시적 후순위/제외** | 화자가 직접 후순위로 명시 | 나중에, 있으면 좋고, 추후 적용, 이번 릴리스 제외 |
| **F. 미결/블로커 (독립 축)** | 정책·수치가 아직 결정되지 않음 | 미정, 논의 중, 결정필요, 더 봐야할듯, (오탈자) 결졍/안남 |

**F축은 우선순위 판정에 쓰지 않는다.** F축은 "TODO로 뺄지 여부"만 결정하는 별도 축이다.

### 판정 원칙 — 복합 판단 (체크리스트 OR 아님)
A~E축 중 하나라도 해당하면 즉시 등급을 확정하지 않는다. 여러 축이 동시에 겹칠 때 그 조합의
강도를 같이 보고 종합 판단한다. (예: B축 단독으로는 약한 신호("이번 분기")여도 D축(요청 증가)과
상황적 긴급성(긴급 소집)이 함께 있으면 종합적으로 강한 신호로 판단)

### 핵심 플로우(조건3) 판정 기준
"서비스 핵심 플로우 차단 · MVP 진입점" 여부는 원문에 **"MVP 범위 제외" / "이번 릴리스 제외" 같은
명시적 제외 문구가 없는 이상 포함으로 간주**한다.

### 등급별 대표 판정 기준
- 🔴 **P1**: A축 / B축과 직결된 핵심 기능 / 조건3-핵심 플로우(제외 문구 없음) / 복합 신호(D+B+긴급성 등 결합)
- 🟠 **P2**: D축 / C축 / 기능 자체는 확정되었고 세부 정책만 F축으로 미결인 경우
- ⚪ **P3**: E축(명시적 후순위)이 문맥상 뚜렷한 **하위 항목** 단위 — 케이스 전체 등급을 끌어내리지 않음

## 2. 이진화 규칙 — P1/P2/P3 → 이번 스프린트급 / 다음 스프린트급

```
이번 스프린트급 = 위 기준 P1
다음 스프린트급 = 위 기준 P2 또는 P3 (구분 없이 통합)
```
단, **P3(E축 명시적 후순위/제외)로 판정된 항목은 "다음 스프린트급"에도 포함하지 않고, Blocker/Warning
계산 대상에서 완전히 제외한다.** (이미 결정·제외된 사항이라 리스크 계산 대상이 아님)

## 3. 신규 축 — G축 (구현 리스크 / 재작업 비용)

Warning 조건 중 "이번 스프린트 변경값이 많아 재정적 문제를 부를 경우"를 판정하기 위한 축.

### G축 하위 기준
| 하위 기준 | 신호 |
|---|---|
| 아키텍처/구현 방식이 완전히 상반되는 모순(Conflict) 존재 | 원문에 "구현 방식이 완전히 달라짐/방식이 갈림" 등 |
| 외부 시스템·솔루션 의존으로 변경 시 비용 발생 | "외부 솔루션 도입 vs 자체개발", "연동" 등 |
| 신규 기술/전문 영역 요구 (사전 등록 키워드에 한정, AI 즉석 추론 금지) | 다국어, AI/추천, 음성인식, OCR, 생체인증, 색약(색상 접근성) 등 — **사전에 등록된 키워드만 인식** |

### G축 판정 원칙 — 키워드 단독 확정 금지, 문맥과 결합해서 종합 판단
- 기술 키워드 감지 + E축(스코프 제외/후순위) 동반 → 리스크 아님, Blocker/Warning 대상 제외
  (예: "색약 모드" + "이번 릴리스 범위에서 제외")
- 기술 키워드 감지 + F축(미결) 또는 Conflict 동반 → 진짜 재작업 리스크로 확정
  (예: "생체인증" + "외부업체 미정/사내 논의 중")
- 기술 키워드 감지 + 방향은 확정되고 일정만 급함(B축) → 리스크보다 일정 압박으로 취급

## 4. Blocker / Warning 최종 연산

```
🚨 Blocker = (이번 스프린트급 스펙) AND (F축 미정 OR Conflict)
             — 고객사 이해관계자 피드백이 반영된 MVP 스펙인데 정책/수치 미정이거나 상반된 요구 충돌
             — 출시 지연 직결 요인, 당장 개발 착수 불가

⚠️ Warning = (다음 스프린트급 스펙) AND (F축 미정 OR 예외 처리 정책 부재 OR G축=리스크 확정)
             — MVP v.2 스펙 중 미결 사항이 있거나 예외 정책이 비어있는 경우
             — 또는 이번 스프린트의 변경값이 커서 재정적 문제를 부를 경우(G축)
```

### 판정 시 유의
- P1(이번 스프린트급) + Conflict 조합은 항상 Blocker (P등급을 낮춰서 완화하지 않음)
- P2(다음 스프린트급) + Conflict 조합은 Warning으로 유지하되, G축 근거를 반드시 판단 근거에 명시
- P3(E축)로 계산 대상에서 제외된 항목은 이 매트릭스 산출 범위 밖 (별도 문서/섹션에서 다룰 사안)
