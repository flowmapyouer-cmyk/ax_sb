## 1. 문제의식 — 왜 단순 키워드 매칭으로는 부족한가

해커톤 원 기준(참고자료 2)은 "꼭/반드시" vs "나중에/있으면 좋고" 두 키워드 축만으로 P1/P2/P3를 나누라고
한다. 그런데 15케이스 원문을 통합해서 보면, 실제 신호는 이 두 키워드 축 바깥에도 존재한다
(예: 케이스 12·13은 키워드 없이도 "긴급 미팅", "사장님 지시"처럼 강한 우선순위 신호를 담고 있음).
키워드 단일 매칭만 쓰면 13/15케이스가 "판단 불가"로 몰려 매번 질문해야 하는 비현실적인 파이프라인이 된다.

## 2. 우선순위 시그널 6대 축

| 축 | 정의 | 예시 키워드/신호 |
|---|---|---|
| **A. 명시적 긴급 키워드** | 화자가 직접 급함을 표현 | 꼭, 반드시, 필수, 즉시 |
| **B. 외부 일정/데드라인** | 서비스 외부 일정과 결부 | 이번 스프린트, 데모 다음주, 이번 분기, 긴급 미팅 |
| **C. 이해관계자 권위/출처** | 발화자의 직책·전문성 | 사장님/경영진 지시, 대표 직접 요청, 보안 컨설턴트 권고 |
| **D. 수요/불만 볼륨** | 요청·불만의 양적 신호 | VoC 많음, 불만 다수, 요청 증가 |
| **E. 명시적 후순위/제외** | 화자가 직접 후순위로 명시 | 나중에, 있으면 좋고, 추후 적용, 이번 릴리스 제외 |
| **F. 미결/블로커 (독립 축)** | 정책·수치가 아직 결정되지 않음 | 미정, 논의 중, 결정필요, 더 봐야할듯, (오탈자) 결졍/안남 |

**F축은 우선순위 판정에 사용하지 않는다.** F축은 "TODO로 뺄지 여부"만 결정하는 별도 축이다.
우선순위(언제 할지)와 미결 여부(뭐가 안 정해졌는지)는 서로 다른 질문이므로 혼용하면 안 된다.

## 3. 판정 원칙 — 복합 판단 (체크리스트 OR 아님)

초안에서는 "A~E축 중 하나라도 해당하면 즉시 등급 확정"하는 체크리스트형 OR 로직으로 설계했으나,
**여러 축이 동시에 겹칠 때 그 조합의 강도를 같이 봐야 한다**는 점이 확인되어 폐기했다.
(예: 케이스 12는 B축 단독으로는 약한 신호("이번 분기")지만, D축(요청 증가)과 상황적 긴급성(긴급 소집)이
함께 실려 있어 종합적으로 P1급 신호로 판단함. 이걸 축마다 독립적으로 쪼개서 판정하면 왜곡된다.)

### 핵심 플로우(조건3) 판정 기준
"서비스 핵심 플로우 차단 · MVP 진입점" 여부는 원문에 **"MVP 범위 제외" / "이번 릴리스 제외" 같은
명시적 제외 문구가 없는 이상 포함으로 간주**한다. (예: 케이스 06 결제 재시도는 제외 문구가 없으므로
핵심 결제 플로우로 포함 → P1. 반대로 케이스 12의 색약 모드는 "이번 릴리스 범위에서 제외"가 명시되어
있어 제외 대상으로 처리.)

### 등급별 대표 판정 예시

- 🔴 **P1**: A축(케이스 02, 14) / B축과 직결된 핵심 기능(케이스 02) / 조건3-핵심 플로우, 제외 문구 없음
  (케이스 01, 06) / 복합 신호(D+B+긴급성, 케이스 12)
- 🟠 **P2**: D축(케이스 04, 08) / C축(케이스 13) / 기능 자체는 확정되었고 세부 정책만 F축으로 미결인 경우
  (케이스 03, 05, 07, 09, 10, 11, 15)
- ⚪ **P3**: E축(명시적 후순위)이 문맥상 뚜렷한 **하위 항목** 단위 (케이스 01의 기간 필터/주별 단위,
  케이스 12의 색약 모드) — 케이스 전체의 등급을 끌어내리지 않고, 해당 하위 항목만 후순위로 별도 처리.

## 4. 모순(Conflict) 케이스 처리 — 별도 리스크 등급 없음

케이스 14·15처럼 상반된 요구가 있는 경우, "블로커/주의 등급"과 같은 별도 리스크 격상 로직은
**적용하지 않는다.** 그 개념은 실제 운영 중인 서비스/대시보드에서 필요한 리스크 관리이며, 이번
산출물(PRD 생성 스킬)의 범위는 "어떤 기능이 있고 그 기획 의도가 뭔지"까지이기 때문이다.
대신 기존 방식 그대로 유지한다: 본문(기능 정의)에서 한쪽으로 단정하지 않고, 미결 항목(TODO)에
A안/B안을 병기한 뒤 의사결정을 요청한다.

## 5. 출력 형식 — 우선순위 섹션과 TODO 섹션의 역할 구분

같은 케이스 안에서 우선순위와 TODO 섹션에 유사한 내용이 겹쳐 보일 수 있는데, 이는 정상이며
제거 대상이 아니다. 두 섹션의 목적이 다르기 때문이다.

| 섹션 | 목적 | 형식 |
|---|---|---|
| **우선순위** | 무엇을 하기로 했는지 (기능 + 등급) | `P{1/2/3} — 근거: "{원문 인용}" (판정축: {축})` |
| **미결 항목(TODO)** | 그 기능 안에서 아직 안 정해진 구체적 질문 | `[TODO] {대상}에게 {무엇} 확정 문의 (A안 vs B안)` 또는 `[TODO] {무엇} 확인 필요` |

예: 케이스 12(다크모드)
- 우선순위: "다크모드 기능 추가 (P1)"
- TODO: "다크모드 적용 범위 확정 문의 (전체 화면 vs 일부 화면만)"

## 6. 질문(사용자 확인) 규칙과의 관계

Operational Rule 2(판단 근거 부족 시 사용자에게 먼저 질문)는 **우선순위 판단에만 한정**해서
적용한다. 위 6대 축 복합 판단으로도 등급을 정할 수 없는 케이스만 모아 한 번에 일괄 질문하고,
그 외 정책·수치 등 F축 미결 항목은 질문 없이 그대로 TODO로 처리한다. v2 데이터 기준으로는
6대 축 복합 판단을 적용한 결과 15케이스 모두 질문 없이 등급을 확정할 수 있었다.
