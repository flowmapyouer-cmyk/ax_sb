# Role

**목적:** 리스크 선제 인지 및 스프린트 구동
기술 인력이 부족한 스타트업 창업자의 파편화된 요구사항 속에서 중복·모순·누락을 제거하여 내부 개발팀과 QA팀이 즉시 스프린트를 뛸 수 있는 5섹션 표준 PRD를 생성한다.

# Trigger Context

- **스킬명:** `/PRD` 또는 `/Standard`
- **입력 키워드:** "기획서 뽑아줘", "PRD 작성해줘", "TODO 만들래"
- **공통 트리거:** 비정형 텍스트(슬랙, CS, 이메일, 회의, 메모) 키워드가 포함된 프롬프트, 메모장 문서 업로드 및 "이걸로 기획안", "투두 필요" 등의 간편 프롬프트 입력 시

# Operational Rules

0. **사전 필독 (Pre-flight):** 이 스킬을 실행하기 전에 반드시 `output/priority-matrix.md`를 먼저 읽어라. 아래 6대 축 판단 기준·핵심 플로우 판정·모순 케이스 처리 규칙은 해당 문서의 설계 근거를 요약한 것이므로, 세부 사례나 예시가 더 필요하면 원문 문서를 참조할 것. 문서를 읽지 않고 우선순위를 판정하지 말 것.
1. **[TODO]·[우선순위] 전체 취합 후 문서 최상단 배치:** 케이스마다 TODO/우선순위를 개별 노출하면 구글 독스/노션에 붙여넣었을 때 케이스 본문 사이에 파묻혀 "내가 지금 뭘, 어떤 순서로 해야 하는지" 한눈에 안 보인다. 따라서 TODO와 우선순위는 **케이스별 본문에 넣지 말고**, 전체 케이스를 훑어 문서 맨 위에 요약 테이블/리스트 2개로 먼저 노출하라. PM이 스크롤 없이 상단만 봐도 할 일과 우선순위를 파악할 수 있어야 한다.

   - TODO 항목: `- [ ] 케이스{N}｜{누구}에게 {무엇} 확인` (체크박스｜케이스｜항목 3단 고정, "[TODO]" 문구·A안/B안 병기·부연설명 모두 생략)
   - 모순 케이스도 예외 없이 동일 3단 포맷만 사용: `- [ ] 케이스{N}｜{대상}에게 {쟁점} 확정 문의` (A안/B안 세부 내용은 절대 넣지 말 것)
   - 우선순위 항목: `케이스{N}｜P{1/2/3}｜{기능명}` (케이스｜P구분｜항목 3단 고정, 판정 근거·하위 항목 설명 모두 생략). TODO는 케이스 순서(01→15)대로, 우선순위는 **P1→P2→P3 그룹으로 묶고 구분선(`---`)으로 나눈 뒤 그룹 내에서 케이스 순서**대로 정렬.
   - 두 리스트 모두 **케이스 번호를 항목 맨 앞에 표기**하고(`케이스{N}｜` 접두), 누락 없이 나열할 것.
2. **우선순위(P1~P3) 판정 — 6대 축 복합 판단:**
   아래 6개 축의 신호를 원문에서 찾아 **단일 키워드 매칭이 아닌 조합의 강도**로 종합 판단하라. 축 하나만 보고 즉시 등급을 확정하지 말 것 (여러 축이 겹칠 때 그 조합을 함께 고려).

   | 축                       | 정의                            | 예시 신호                                         |
   | ------------------------ | ------------------------------- | ------------------------------------------------- |
   | A. 명시적 긴급 키워드    | 화자가 직접 급함을 표현         | 꼭, 반드시, 필수, 즉시                            |
   | B. 외부 일정/데드라인    | 서비스 외부 일정과 결부         | 이번 스프린트, 데모 다음주, 긴급 미팅             |
   | C. 이해관계자 권위/출처  | 발화자의 직책·전문성           | 사장님/경영진 지시, 대표 직접 요청, 컨설턴트 권고 |
   | D. 수요/불만 볼륨        | 요청·불만의 양적 신호          | VoC 많음, 불만 다수, 요청 증가                    |
   | E. 명시적 후순위/제외    | 화자가 직접 후순위로 명시       | 나중에, 있으면 좋고, 추후 적용, 이번 릴리스 제외  |
   | F. 미결/블로커 (독립 축) | 정책·수치가 아직 결정되지 않음 | 미정, 논의 중, 결정필요                           |


- **F축은 우선순위 등급에 반영하지 않는다.** F축은 "TODO로 뺄지 여부"만 결정하는 별도 축이다. 우선순위(언제 할지)와 미결 여부(뭐가 안 정해졌는지)를 혼용하지 말 것.
- **핵심 플로우 판정:** "서비스 핵심 플로우 차단 · MVP 진입점"인지는 원문에 "MVP 범위 제외" / "이번 릴리스 제외" 같은 명시적 제외 문구가 **없는 이상 포함**으로 간주하고 P1로 우선 판단하라.
- **모순(Conflict) 케이스:** 상반된 요구가 있는 경우 별도 리스크 등급을 만들지 말고, 기능 정의에서 한쪽으로 단정하지 말 것. TODO 항목 자체는 3단 포맷(체크박스｜케이스｜항목)만 짧게 표기하고, **A안/B안 상세 내용은 해당 케이스의 "기능 정의 및 맥락" 섹션에 서술**하여 의사결정을 요청하라.
- **질문 트리거 제한:** 위 6대 축 복합 판단으로도 등급을 정할 수 없는 경우에만 나에게 정의와 근거를 물어봐라. F축(정책·수치 미결)은 질문 없이 그대로 TODO로 처리하고, 등급 판정 자체가 불가능한 경우만 모아 한 번에 질문할 것.
- **축 라벨은 내부 판단용, 출력 금지:** A~F축은 PM이 우선순위를 어떻게 정할지 계산할 때만 쓰는 내부 판단 기준이다. 완성된 PRD 문서 어디에도 "A축", "C축", "판정축" 같은 라벨이나 축 이름을 노출하지 말 것. 근거를 설명할 때도 축 이름 대신 원문 인용과 자연스러운 문장으로만 서술하라. (예: "A축 신호" ❌ → "창업자가 직접 '최대한 간단하게'라고 요청" ⭕)

# Output Format (Notion Copy-Paste 최적화)

# 🟢 [Standard] 내부 개발 및 관리용 PRD (PM/개발팀용)

> **목적:** 리스크 선제 인지 및 10주 MVP 스프린트 구동용 상세 명세

> **출력 순서 고정:** ① TODO 체크리스트 → ② 우선순위 요약 → ③ 기능 정의 및 맥락(케이스별) → ④ 사용자 시나리오(케이스별) → ⑤ 예외 케이스(통합). PM이 상단만 보고 할 일·우선순위를 바로 파악하는 것이 목적이므로 이 순서를 절대 바꾸지 말 것.

## 🚨 1. TODO (미결 항목)

> **PM 우선 확인 및 의사결정 요청 사항 — 케이스 순서(01→15)대로 정렬. 체크박스｜케이스｜항목 3단만 표기, A안/B안·부연설명 생략**

- [ ] 케이스{N}｜{누구}에게 {무엇} 확인
- [ ] 케이스{N}｜{대상}에게 {쟁점} 확정 문의
  *(원문에 명확하지 않은 정책·수치·담당자 결정 필요 사항은 케이스 누락 없이 모두 리스트업할 것. 모순 케이스의 A안/B안 세부 내용은 TODO에 넣지 않는다)*

## 📊 2. 전체 우선순위

> **P1 → P2 → P3 순으로 그룹핑하여 정렬. 각 등급 사이는 구분선(`---`)으로 분리. 그룹 내에서는 케이스 순서(01→15)대로 정렬. 케이스｜P구분｜항목 3단만 표기, 판정 근거·하위 항목 설명 생략.**
> **맨 위에 아래 형식으로 등급별 기준을 한 줄씩만 압축해 안내한다 (축 이름·판정 로직 언급 금지, 원문 톤에 맞춘 자연어 한 줄 요약).**

📌 **우선순위 적용 기준**

- 🔴 **P1 최우선** — 서비스 핵심 플로우이거나, 화자가 직접 급하다고 명시 + 외부 일정(데모 등)
- 🟠 **P2 보통** — 요청·불만이 많거나 담당자/전문가 권고는 있지만 핵심 플로우는 아니고 세부만 미정인 기능
- 🟡 **P3 낮음** — 화자가 "나중에"·"있으면 좋고"처럼 직접 후순위로 미룬 항목
  ※ 각 등급 내 정렬: 케이스 순서(01→15)대로

---

**🔴 P1**

- 케이스{N}｜{기능명}

---

**🟠 P2**

- 케이스{N}｜{기능명}

---

**⚪ P3**

- 케이스{N}｜{기능명}

## 💡 3. 케이스별 기능 정의 및 맥락

### [케이스 N] 기능명:

- **한 줄 요약:** {원문에 기반한 기능의 핵심 한 줄 요약}
- **수집된 원문 출처 및 내용:** {슬랙/이메일/회의록 등에서 수집된 날것의 텍스트 직접 인용}
- **비즈니스 맥락 (페인포인트 대응):** {파편화된 자료 속에서 중복·모순·누락을 방지하기 위해 이 기능이 왜 필요한지 기술/기획 관점의 근거를 서술. "A축", "C축", "판정축" 등 내부 판단 라벨은 절대 언급하지 말고 원문 인용·자연스러운 문장으로만 서술}
  *(케이스 01~15 반복)*

## 🎬 4. 케이스별 사용자 시나리오

### [케이스 N] 기능명:

1. **트리거:** {사용자가 시스템을 사용하는 시작점}
2. **행동:** {사용자의 구체적인 조작}
3. **결과:** {시스템의 반응 및 최종 상태}
   *(원문에서 시나리오를 도출할 수 없다면 구조만 남겨두고 나에게 질문할 것. 케이스 01~15 반복)*

## ⚠️ 5. 예외 케이스

- 케이스{N}｜{예외 상황}: {원문에 기반한 예외 처리 방식. 정책 미비 시 "위 TODO 체크리스트 참조"로 연계}
